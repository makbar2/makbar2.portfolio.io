PharmaFriends

Project Summary

PharmaFriends was a platform designed to help pharmacists conduct NMS (New Medicine Service) consultations remotely. It was developed for a startup aiming to handle these consultations on behalf of independent pharmacies.

Tech Stack

  • Node.js
  • Express.js
  • MySQL
  • JWT Authentication
  • Twig templating

Role & Contributions

I joined the project after the initial prototype was created. My work included:

  • Modularizing the backend and restructured the database in 3NF to improve maintainability and reduce redundancy.
  • Refactored legacy code and HTML, enhancing scalability, readability, and long-term maintainability.
  • Implemented JWT-based role authentication and usage analytics for pharmacists and pharmacy managers.
  • Implemented an NMS consultation form for pharmacists to enter consultation results
  • Designed a dynamic admin interface to allow non-technical users to audit and modify data safely without direct database access.
  • Developed a pharmacy dashboard to track NMS consultation progress and facilitate new submissions efficiently.

Decisions and Challenges

Scope Creep

Main challenge for developing this program was requirements gathering, as some things weren't made clear enough or features were added that weren't needed. One instance of this is I implemented session timing for consultations (tracking start and end times) based on early discussions about potentially recording audio and needing accurate logs for when consultations were completed, this wasn't needed and I had to remove a few hours worth of work. This taught me the importance of confirming the scope early.

Database

I decided to implement the database handler as a static class, since the database is used by all the endpoints. Making the class static ensures a single, persistent connection logic stays in memory for the life of the server process.

Admin page

To ensure a smooth user experience without page reloads, I built all admin tools into a single page. Each tool was separated into its own Twig file and hidden by default with css. When an admin selects a tool, it's shown dynamically by toggling the visibility class.

Outcomes & Lessons learned

This project highlighted the importance of clear communication with clients/stakeholders early detailed discussions about each feature could have saved significant time. Developing this system provided me with valuable experience in working with an existing codebase, designing scalable backend systems, and managing real world product ambiguity.
